# Security Model

This document reflects the currently implemented security controls in AUFlow.

## Authentication Surface

Active auth routes support:

- login/logout
- forgot/reset password
- confirm password
- forced password change (`password/change`)

Registration and email verification routes are not active in the current route map.

## Forced Password Change

- New users are created with `must_change_password=true`.
- `ForcePasswordChange` middleware redirects authenticated users with that flag to `password.change` unless route is allowlisted.
- Password reset and password change flows clear the flag.

## Authorization Model

- Authorization is permission-slug based.
- Middleware aliases:
  - `permission`
  - `any-permission`
- Role escalation safeguards exist in `RolePolicy` for globally sensitive permission slugs.
- Resource policies protect user and role management flows.

## Request Hardening

### Throttling

- Submission routes use `throttle:submissions`.
- Submission limiter is configured in `AppServiceProvider`:
  - 5 requests/minute
  - keyed by `account_id` when authenticated, else client IP.
- Public error-report submission endpoint is throttled (`10,1`).
- Forgot-password endpoint is throttled (`5,1`).

### Path Sanitization

- File-serving endpoints normalize and validate paths to block traversal (`../`) patterns.
- `/files/{path}` and `/profile-pictures/{path}` both apply path normalization and existence checks.

## Data Protection

### Canonical Submission Payload Storage

- `FormSubmission.payload_json` is stored as plain JSON (no application-level encryption).
- Access control is enforced at the database and application authorization layer.
- Form-level `sensitive_fields` metadata is available on the form schema for UI-level field handling.

### Snapshot Integrity Signing

- Snapshots store `action_hash` generated by HMAC-SHA256 over actor/timestamp/payload.
- Signing key source: `workflow.snapshot.signing_key` (`SNAPSHOT_SIGNING_KEY`, fallback `APP_KEY`).
- Optional legacy hash verification support is config-driven (`SNAPSHOT_ALLOW_LEGACY_HASH_VERIFICATION`).

## Immutability Controls

### Audit Log

- `tbl_audit_log` update/delete operations are blocked by database triggers.

### Snapshot

- DB trigger blocks updates when `locked=1`.
- Model guard throws `SnapshotImmutableException` if code attempts to save locked records.

## File and Media Security

- Profile pictures are stored on private `profile-pictures` disk.
- Access is mediated through authenticated controller route, with legacy public fallback only when needed.
- Submission and progress attachments are generally served from protected routes and permission-checked contexts.

## Error Disclosure Controls

- Exception handling in `bootstrap/app.php` returns safe messages in non-local environments.
- Inertia error page handling is used for major HTTP error classes on SPA requests.
- Raw exception details are not exposed in production responses.

## Operational Security Notes

- `APP_DEBUG` should be false in production.
- `SNAPSHOT_SIGNING_KEY` must be managed as a secret and rotated with migration planning.
- Production deployment scripts intentionally avoid destructive DB commands.
