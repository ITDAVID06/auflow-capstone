# Workflow Engine

This document explains AUFlow's current workflow runtime implementation.

## Design-Time vs Runtime

## Design-Time

Administrators maintain workflows through workflow management and canvas endpoints (`/workflows/*`).
Design-time data is stored in:

- `tbl_workflow`
- `tbl_workflow_step`
- `tbl_workflow_step_approvers`

Design-time records are mutable while in draft and can be revised/duplicated/archived.

## Runtime

Runtime progression is based on published workflow versions:

- `tbl_workflow_version` contains frozen `steps_snapshot` JSON.
- New submissions bind to `workflow_version_id`.
- Initial progress rows are derived from snapshot data, not from mutable step rows.

## Publish and Version Snapshot

`WorkflowService::publishWorkflow()`:

1. acquires lock on workflow row
2. validates publish prerequisites
3. archives sibling revisions where needed
4. snapshots current steps/approvers into `tbl_workflow_version.steps_snapshot`
5. marks previous versions `is_current=false`
6. creates new `is_current=true` version row
7. activates workflow and syncs bound form state

## Submission Bootstrap

On submission, services build payloads and dispatch `ProcessFormSubmissionJob`.
The job calls `WriteCanonicalSubmissionAction`, which writes canonical records transactionally and persists workflow progress payload rows.

`WorkflowProgressService::buildInitialProgress()` is snapshot-first and supports:

- group-based pending/waiting initialization
- condition-based initial step skipping
- assignment/approver resolution from snapshot

## Approval and Rejection Processing

### Staff Approval/Rejection

`StaffSubmissionService` paths (`approveStep`, `rejectStep`) use:

- `DB::transaction()`
- `lockForUpdate()` on `tbl_workflow_step_progress`
- `lockForUpdate()` on parent `tbl_form_submission`

Behavior includes:

- actor authorization checks (assigned actor or valid approver)
- status updates + timing fields
- optional progress comment attachments
- snapshot generation dispatch (`dispatchAfterResponse`)
- group-aware workflow advancement

### Admin Override

`AdminSubmissionsOverrideService::adminOverride()` uses the same locking strategy and handles:

- override approve/reject status transitions
- downstream and peer-step rejection cascades on reject
- workflow advancement on approve
- completion notifications
- snapshot generation dispatch

## Group Advancement Logic

Workflow advancement uses `step_group` semantics.
For completed groups, waiting rows in the next group transition to pending.

Current implementation is snapshot-first, with explicit fallback to legacy live-step logic for submissions that pre-date workflow version binding.

## Runtime Conditions and Skips

Condition evaluation supports:

- watch-field emptiness checks
- branch-condition evaluation through `WorkflowConditionEvaluator`

Skip outcomes are persisted as `Skipped` status in progress rows.

## Notifications Integration

Workflow transitions call notification services for:

- next approvers in sequence/group
- completion outcomes (`approved`/`rejected`)

Reminder scheduling is command-driven (`workflow:send-approval-reminders`) and reads pending progress state.

## Snapshot Integration

Per-step snapshots are generated by `GenerateVerificationSnapshot` job and built by `SnapshotService`.

Snapshot payload includes:

- top-level status and workflow-step labels
- form/submission metadata
- approval details
- ordered fields
- attachment metadata
- approval history snapshot
- `is_workflow_complete` flag

Integrity is verified using HMAC (`SnapshotSecurityService`) with `SNAPSHOT_SIGNING_KEY`.

## Legacy Compatibility Paths

Compatibility code remains for pre-version submissions:

- fallback to live `tbl_workflow_step` queries when version snapshot data is unavailable
- optional legacy hash verification for snapshots (config-driven)

These paths are present for historical data safety, while new runtime flow is version-snapshot based.
